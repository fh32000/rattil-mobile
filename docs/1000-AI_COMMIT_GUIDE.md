# AI Agent Commit Guide

This document provides instructions and best practices for AI agents on how to properly structure, group, and execute Git commits within this project.

## 1. Commit Message Structure

All commit messages MUST follow the Conventional Commits format:

```text
<type>: <subject>
```

### Allowed Types:
- **`feat`**: A new feature or structure update.
- **`fix`**: A bug fix.
- **`docs`**: Documentation only changes.
- **`style`**: Changes that do not affect the meaning of the code (white-space, formatting, etc.).
- **`refactor`**: A code change that neither fixes a bug nor adds a feature (e.g., renaming variables, simplifying logic).
- **`perf`**: A code change that improves performance.
- **`test`**: Adding missing tests or correcting existing tests.
- **`chore`**: Changes to the build process or auxiliary tools and libraries.

*Example:* `feat: add user authentication module`

## 2. The Commit Workflow

When asked to commit changes, **DO NOT** commit all modified or unstaged files at once using commands like `git commit -am "..."` or `git add .` followed by a single commit, unless the changes are extremely small and strictly related.

Follow this systematic workflow:

1. **Inspect the Changes**:
   - Check the status of the repository (`git status`).
   - Review the actual code changes (`git diff` or `git diff <file>`).
   - Understand what modifications have been made.

2. **Group Related Changes**:
   - Group the modified files, or even partial modifications within a single file (using `git add -p`), based on their logical purpose.
   - Separate different types of work. For example, if you see a bug fix, a documentation update, and a performance improvement, these MUST be split into three distinct groups.

3. **Stage and Commit**:
   - Stage the first group of related changes.
   - Write a clear, concise, and accurate commit message for this specific group following the structure above.
   - Commit the staged changes.
   - Repeat this process for the remaining groups of changes.

## 3. Strict Rules & Best Practices

- **Avoid Monolithic Commits**: Never group a large number of disparate, unrelated modifications into a single commit.
- **Clear Separation of Concerns**: Documentation updates, structure updates, code logic addtions, bug fixes, performance improvements, and file/variable renaming **must** all be separated into their own respective commits.
- **Clarity is Key**: Make sure that the commit message accurately reflects *only* the changes included in that specific commit.
- **Partial Commits**: If a single file contains both a bug fix and a new feature, you must stage and commit the bug fix separately from the new feature (e.g., using `git add --patch`).
- **Exclude Temporary or AI-Generated Files**: Even if explicitly asked to "commit all changes", do **NOT** commit temporary files, scratchpads, research reports, standalone test scripts, or any files generated by the AI agent for its own deduction that are not part of the project framework. If you omit any such files, explicitly state in your response report to the user that they were excluded because they are temporary and unrelated to the actual project development.

## 4. Branching Strategy — GitFlow (CRITICAL)

This project follows the **GitFlow** branching model. **NEVER commit directly to `main` or `develop` branches.**

### Branch Types:

| Branch | Purpose | Branches from | Merges into |
|--------|---------|---------------|-------------|
| `main` | Production-ready code | — | — |
| `develop` | Integration branch for features | `main` | `main` (via release) |
| `feature/*` | New features & improvements | `develop` | `develop` |
| `docs/*` | Documentation updates | `develop` | `develop` |
| `hotfix/*` | Urgent production fixes | `main` | `main` & `develop` |
| `release/*` | Release preparation & final fixes | `develop` | `main` & `develop` |

### Branch Naming Convention:

```text
feature/<developer-name>/<short-description>
docs/<developer-name>/<short-description>
hotfix/<short-description>
release/<version>
```

- `<developer-name>`: The developer's alias or name (e.g., `fh32000`, `ab1234`).
- `<short-description>`: A brief, kebab-case summary of the work (e.g., `improve-engines`, `add-webhook-logging`).
- `<version>`: Semantic version number for releases (e.g., `1.2.0`).

*Examples:*
- `feature/fh32000/improve-engines`
- `feature/fh32000/add-webhook-logging`
- `docs/fh32000/update-commit-guide`
- `feature/fh32000/save-memorization`
- `hotfix/fix-payment-gateway`
- `release/2.1.0`

### Pre-Commit Checklist & Branch Update Workflow:

Before making any commits, you MUST follow this sequence to ensure your branch is up-to-date and conflicts are handled properly:

1. **Check the current branch** (`git branch --show-current`).
2. **If on `main` or `develop`**: **STOP.** Do NOT commit. You must create or switch to a feature branch.
3. **Stash current changes**: Run `git stash` to safely store your uncommitted modifications. This prevents merge conflicts during the update process.
4. **Update the branch**: Pull the latest changes from the remote repository to ensure the branch is fully updated.
5. **Apply stashed changes**: Run `git stash pop` to bring back your modifications.
6. **Handle Conflicts**:
   - **No conflicts**: Excellent, proceed to commit.
   - **Simple conflicts**: Try to resolve them intelligently and document the resolution in your final report to the user.
   - **Difficult conflicts**: STOP and review the conflicts with the developer. Ask them how to proceed to solve the problem.

### Multi-Feature Chaining Strategy:

If you are asked to implement multiple scattered modifications that should logically be separated into more than one feature branch:

1. **Start from `develop`**: Checkout the `develop` branch, update it to fetch the latest data, and create your first feature branch (e.g., `feature/<developer-name>/<feature-1>`).
2. **Commit Feature 1**: Add and commit the appropriate modifications for the first feature.
3. **Chain the Next Feature**: To avoid returning to `develop` and to maintain the flow of changes, **continue from where you stopped**. Create the next feature branch directly from the end of the previous one (e.g., while on `feature-1`, run `git checkout -b feature/<developer-name>/<feature-2>`).
4. **Repeat**: Commit changes for feature 2, then branch out for feature 3 from the same place, and so on.
5. **Push and PR**: Finally, push all the created feature branches to the remote repository and create Pull Requests for **all of them** against `develop`.

### If No Suitable Branch Exists:

- **ASK the user** before creating or switching branches (unless applying the Chaining Strategy above). Propose:
  1. Create a new `feature/<developer-name>/<description>` branch off `develop`.
  2. Switch to an existing branch if one is already open for the related work.
- **Wait for confirmation** before proceeding.

### Tags:

After merging a `release/*` or `hotfix/*` branch into `main`, a version tag should be created:

```text
v<major>.<minor>.<patch>
```

*Example:* `v2.1.0`

---

By following these instructions, you ensure that the project's Git history remains clean, readable, and easy to revert or track changes over time.
